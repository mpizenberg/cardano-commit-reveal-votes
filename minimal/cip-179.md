## Abstract

This proposal defines a standardized transaction metadata format for creating, responding to, and cancelling on-chain surveys and polls under metadata label `17`.

The format supports:
- One or more survey definitions per transaction.
- One or more survey responses per transaction.
- Seven built-in question types (custom, single-choice, multi-select, ranking, numeric-range, points-allocation, rating) via a tagged sum type.
- Optional linkage to governance Info Actions via anchor metadata.
- Deterministic response binding using a survey reference `(TxId, index)` pair.
- Survey cancellation by the original creator.
- Public or timelocked responses, the latter using Drand `tlock` for delayed reveal.
- Optional off-chain content anchors (URI + hash) for bulky presentation text, custom-method schemas, and voter rationales.
- Eligibility expressed as a set of roles; result weighting and aggregation are deliberately left out of scope.

The standard is general-purpose and can be used for governance and non-governance sentiment gathering.

## Motivation: Why is this CIP necessary?

Formal governance actions are intentionally constrained. Those constraints are useful for protocol safety, but they are not sufficient for many community workflows that need structured sentiment data. This is particularly true for Info Actions, which are limited in their ability to collect information from stakeholders.

Examples include:
- Polling candidate CIPs to decide hard-fork prioritization.
- Voting on a line item of a budget proposal.
- Gathering bounded numeric preferences (for example, initialization values for several related parameters).
- Ranking candidates or proposals by community preference.

Without a shared on-chain format, these workflows fragment across custom off-chain tools and incompatible schemas. This CIP provides a common metadata interface that wallets, explorers, governance dashboards, and indexers can implement consistently.

## Specification

### Overview

This CIP reserves metadata label `17` for three payload types:
- Survey definitions (tag `0`): one or more survey definitions.
- Survey responses (tag `1`): one or more survey responses.
- Survey cancellations (tag `2`): one or more survey cancellations.

A transaction MUST include at most one label `17` payload. The three payload types MUST NOT be mixed in a single transaction.

Survey metadata under label `17` is valid as a standalone mechanism and does not require any governance action.

A survey is identified by a `survey_ref`: the pair `(tx_id, survey_index)` where `tx_id` is the transaction containing the survey definitions payload and `survey_index` is the position of the definition within that payload's definitions array. This follows the Cardano convention used for UTxOs (`transaction_input`), governance actions (`gov_action_id`), and other on-chain references.

### Encoding Conventions

All map keys and enumeration values use integers for compact CBOR encoding, consistent with how the Cardano ledger encodes transaction bodies, certificates, and voting procedures.

The two top-level records, `survey_definition` and `survey_response`, are encoded as integer-keyed CBOR maps rather than positional arrays. This follows the Conway `transaction_body` pattern: optional and future fields live at distinct keys, so a later version can add fields without renumbering and a decoder can skip keys it does not recognize. The lower-level discriminated unions (`survey_question` and `answer_item` variants and their nested structures) remain tagged positional arrays — they are versioned by tag/URN and evolve by adding new tags, not new fields, so a map would only bloat them.

Encoders MUST use deterministic CBOR for these maps: integer keys sorted in ascending numeric order, with no duplicate keys. (For the small integer keys used here this is plain numeric order, per RFC 8949 §4.2.) Deterministic encoding keeps any hashing, equality, or deduplication over the payload unambiguous.

Text fields that may exceed the 64-byte Cardano metadata text limit use chunked text: either a single `bounded_text` string (when it fits within 64 bytes) or an array of text strings, each at most 64 bytes, concatenated to reconstruct the full value. The array form follows the same chunking approach used by CIP-20. Implementations MUST accept both forms.

Transaction IDs and hashes use raw `bytes .size 32` rather than hex-encoded text strings, halving the per-hash cost.

Bulky human-readable text (titles, descriptions, question prompts, option labels) MAY be moved off-chain entirely, referenced from on-chain by a `content_anchor` (URI + `blake2b-256` hash). See [External Survey Definitions](#external-survey-definitions).

Naming convention: on-chain (CBOR) fields and protocol-level identifiers use `snake_case`, matching the Conway ledger CDDL this encoding follows. Keys in off-chain JSON documents (such as the Info Action survey-link anchor) use `camelCase`, matching JSON convention. On the wire, fields are identified by position (in the tagged arrays) or by integer key (in the `survey_definition` / `survey_response` maps), so these names are documentation only, they do not appear in the encoding.

### CDDL Schema

```cddl
; CIP-179 On-chain Surveys and Polls (version 4)
;
; Encoding principles:
;   - Integer keys and enum values for compact CBOR.
;   - Chunked text arrays for strings that may exceed 64 bytes.
;   - Raw bytes for hashes and transaction IDs.
;   - Sum types (tagged unions) for question and answer variants.
;   - (TxId, index) pairs for survey references.
;   - Optional off-chain content anchors (URI + hash) for bulky text.

; ---------- Primitives ----------

pos_uint = uint .gt 0
tx_id = bytes .size 32
blake2b_256 = bytes .size 32
drand_chain_hash = bytes .size 32     ; Drand chain hash for tlock
bounded_text = text .size (0..64)     ; Cardano metadata text limit
chunked_text = bounded_text / [+ bounded_text]  ; single string or chunked array
survey_ref = [tx_id, uint .size 2]    ; (TxId, index in definitions array)

; A reference to off-chain content, tamper-evident via its hash. Used for:
; external survey presentation text, custom-method schemas, and voter rationales.
content_anchor = [
  chunked_text,   ; URI (e.g. ipfs://..., https://...)
  blake2b_256     ; blake2b-256 of the raw bytes at that URI
]

; Standard Cardano transaction metadatum.
transaction_metadatum =
    { * transaction_metadatum => transaction_metadatum }
  / [ * transaction_metadatum ]
  / int
  / bytes .size (0..64)
  / bounded_text

; ---------- Types reused from Conway ledger CDDL ----------
; Included for reference; authoritative definitions are in the ledger spec.

hash28 = bytes .size 28
addr_keyhash = hash28
script_hash = hash28

; A Cardano credential: key-based (tag 0) or script-based (tag 1).
credential = [0, addr_keyhash // 1, script_hash]

; ---------- Top-level payload under metadata label 17 ----------

cip_179_payload = [0, [+ survey_definition]]
                / [1, [+ survey_response]]
                / [2, [+ survey_cancellation]]

; ---------- Survey definition ----------

; Integer-keyed map (not a positional array) so future optional fields can be
; added at new keys without renumbering, mirroring how the Conway ledger encodes
; `transaction_body`. Encoders MUST use deterministic CBOR (see Encoding
; Conventions): integer keys, ascending, no duplicates. Unknown keys are reserved
; for future versions; decoders SHOULD ignore keys they do not recognize.
survey_definition = {
  0 => uint,                   ; spec_version (this document = 4)
  1 => credential,             ; owner (for cancellation authorization)
  2 => chunked_text,           ; title (MAY be empty in external-content mode)
  3 => chunked_text,           ; description (MAY be empty in external-content mode)
  4 => eligible_roles,         ; roles permitted to respond
  5 => uint,                   ; end_epoch (inclusive cutoff)
  6 => submission_mode,        ; how responses are submitted
  7 => [+ survey_question],    ; questions (at least one)
  ? 8 => content_anchor        ; OPTIONAL external presentation document
}

; ---------- Eligible roles ----------

; Roles. A role declares eligibility (which credential type may respond) and
; serves as a UI hint for which key a respondent should present. A role is NOT
; a weighting directive: how a recorded selection is weighted or aggregated is
; out of scope (see "Weighting and Aggregation (out of scope)").
;   0 = DRep
;   1 = SPO
;   2 = CC
;   3 = Stakeholder
;   4 = Owner          (control of a payment/spending credential)
role = 0 / 1 / 2 / 3 / 4

; Roles permitted to respond. Non-empty; entries SHOULD be unique.
eligible_roles = [+ role]

; ---------- Submission mode (tagged sum type) ----------

submission_mode = public_submission_mode / timelocked_submission_mode

; Tag 0: Public. Responses carry plaintext answer items.
public_submission_mode = [0]

; Tag 1: Timelocked. Responses carry a Drand `tlock` ciphertext instead of
;   plaintext answers. Answers are encrypted at submission and become
;   decryptable by anyone once `round` publishes on the pinned Drand chain.
;   This is delayed reveal, not permanent secrecy. `padding_size` is the minimum
;   plaintext byte length each response is padded to before encryption.
timelocked_submission_mode = [1, drand_chain_hash, pos_uint, pos_uint]
                          ;   chain_hash        round     padding_size

; ---------- Question types (tagged sum type) ----------

survey_question = custom_question
                / single_choice_question
                / multi_select_question
                / ranking_question
                / numeric_range_question
                / points_allocation_question
                / rating_question

; Tag 0 is reserved for the custom (extension) type, so new built-in types are
; appended at higher tags without ever disturbing it.
;
; Every question ends with an OPTIONAL `required` flag (bool, default false).
; When true, the question MUST NOT be omitted in a response. Omission of a
; non-required question means abstain (see "Abstain semantics").

; Tag 0: Custom method. Answer format defined by the schema at the anchor.
custom_question = [0, chunked_text, content_anchor, ? bool]

; Tag 1: Single-choice. Exactly one option selected.
single_choice_question = [1, chunked_text, options_or_count, ? bool]

; Tag 2: Multi-select. Between min_selections and max_selections options.
;   0 <= min_selections <= max_selections <= number_of_options.
;   min_selections may be 0: a present, empty selection is a valid answer
;   ("none selected"), distinct from omitting the question (abstain).
multi_select_question = [2, chunked_text, options_or_count, uint, pos_uint, ? bool]
                     ;   tag  prompt        opts/count       min_sel  max_sel

; Tag 3: Ranking. Between min_ranked and max_ranked options, preference order.
;   1 <= min_ranked <= max_ranked <= number_of_options.
ranking_question = [3, chunked_text, options_or_count, pos_uint, pos_uint, ? bool]
                ;   tag  prompt        opts/count       min_rank  max_rank

; Tag 4: Numeric-range. Answer is an integer satisfying the constraints.
numeric_range_question = [4, chunked_text, numeric_constraints, ? bool]

; Tag 5: Points-allocation. Distribute exactly `budget` points across options.
points_allocation_question = [5, chunked_text, options_or_count, pos_uint, ? bool]
                          ;   tag  prompt        opts/count       budget

; Tag 6: Rating. Rate options on the scale given by rating_scale (numeric grid
;   or ordered label list).
rating_question = [6, chunked_text, options_or_count, rating_scale, ? bool]

; ---------- Rating scale ----------

; The scale used by a rating question is either a numeric grid (presented as
; numbers, e.g. 1..5) or an ordered list of level labels from worst to best
; (presented as text, e.g. "bad","average","good"). In external-content mode the
; labels MAY be replaced by a level count (>= 2). A CBOR decoder distinguishes
; the three forms: a numeric grid is an array of ints, a label list is an array
; of text, a count is a bare uint.
;   - numeric grid:  the rating answer is an int value on the grid.
;   - label list:    the rating answer is the 0-based index into the labels.
rating_scale = numeric_constraints / options_or_count

; ---------- Numeric constraints ----------

numeric_constraints = [
  int,              ; min_value
  int,              ; max_value (>= min_value)
  ? pos_uint        ; step (optional, must be > 0)
]

; ---------- Options ----------

; Option-bearing questions carry either inline labels (>= 2) or, in
; external-content mode (the survey_definition carries a content_anchor), an
; option count (>= 2). A CBOR decoder distinguishes the array form (inline
; labels) from the uint form (count) directly.
options          = [2* bounded_text]
option_count     = uint .ge 2
options_or_count = options / option_count

; ---------- Survey response ----------

; spec_version lets a response be decoded (including its answer encoding and
; public-vs-timelocked answer shape) without first resolving the survey.
; Integer-keyed map for the same forward-compatibility reasons as
; survey_definition: the deferred CIP-8 message-signing proof will add optional
; fields (signature material, nonce) alongside the optional rationale, and a map
; expresses several independent optionals cleanly. Same deterministic-CBOR rule.
survey_response = {
  0 => uint,                   ; spec_version (this document = 4)
  1 => survey_ref,             ; reference to the survey definition
  2 => role,                   ; claimed responder role
  3 => credential,             ; responder's credential
  4 => response_answers,       ; answers (an omitted question = abstain)
  ? 5 => content_anchor        ; OPTIONAL voter rationale
}

; Response answers depend on the referenced survey's submission_mode:
;   - Public: an array of plaintext answer items.
;   - Timelocked: a Drand tlock ciphertext (chunked bytes), opaque until the
;     survey's round publishes. Distinguished from the public form because its
;     elements are byte strings rather than arrays.
response_answers = [+ answer_item] / chunked_bytes

; Chunked byte blob: payloads exceeding the 64-byte metadata limit are split
; into an array of <=64-byte byte strings, concatenated to reconstruct.
chunked_bytes = (bytes .size (0..64)) / [+ bytes .size (0..64)]

; ---------- Answer types (tag matches question type) ----------

answer_item = custom_answer
            / single_choice_answer
            / multi_select_answer
            / ranking_answer
            / numeric_answer
            / points_allocation_answer
            / rating_answer

; In all answer variants, the second element is the question index
; (position in the survey_definition's questions array).

; Tag 0: Custom answer.
custom_answer = [0, uint, transaction_metadatum]

; Tag 1: Single-choice answer. Exactly 1 selected option index.
single_choice_answer = [1, uint, uint]

; Tag 2: Multi-select answer. min_selections..max_selections indices.
;   Indices MUST be unique and valid positions in the options. The index list
;   MAY be empty when min_selections is 0 (an explicit "none selected" answer).
multi_select_answer = [2, uint, [* uint]]

; Tag 3: Ranking answer. min_ranked..max_ranked indices, most preferred first.
;   Indices MUST be unique and valid positions in the options.
ranking_answer = [3, uint, [+ uint]]

; Tag 4: Numeric-range answer.
numeric_answer = [4, uint, int]

; Tag 5: Points-allocation answer. (option_index, points) pairs.
;   points >= 0; option indices unique; unlisted options receive 0;
;   sum of points MUST equal `budget`.
points_allocation_answer = [5, uint, [+ [uint, uint]]]

; Tag 6: Rating answer. (option_index, rating) pairs.
;   rating MUST satisfy the rating grid; option indices unique.
rating_answer = [6, uint, [+ [uint, int]]]

; ---------- Survey cancellation ----------

; Cancels a previously published survey.
; The cancellation transaction MUST prove ownership of the
; survey definition's owner credential.
survey_cancellation = survey_ref
```

#### Roles

| Integer | Role | Ledger / proof basis |
|:--------|:-----|:---------------------|

| 1 | SPO | Pool cold credential (calidus hot-key proof is an open design question). |
| 2 | CC | Hot credential of an active Constitutional Committee member. |
| 3 | Stakeholder | Stake credential with delegated stake. |
| 4 | Owner | Control of a payment/spending credential; eligibility (e.g. NFT ownership) is enforced off-chain. |

A role declares eligibility and hints to UIs which key to present. It is not a weighting directive (see [Weighting and Aggregation (out of scope)](#weighting-and-aggregation-out-of-scope)).

#### Payload tags

| Tag | Payload type |
|:----|:-------------|
| 0 | Survey definitions |
| 1 | Survey responses |
| 2 | Survey cancellations |

#### Question and answer type tags

| Tag | Question type | Answer format |
|:----|:--------------|:--------------|
| 0 | Custom | `[0, q_idx, metadatum]` |
| 1 | Single-choice | `[1, q_idx, option_index]` |
| 2 | Multi-select | `[2, q_idx, [option_index*]]` |
| 3 | Ranking | `[3, q_idx, [option_index+]]` |
| 4 | Numeric-range | `[4, q_idx, int_value]` |
| 5 | Points-allocation | `[5, q_idx, [[option_index, points]+]]` |

### Method Identifier Registry

Built-in question types are identified on-chain solely by their integer tag; the encoding stays compact and self-describing. For interoperability with tools and with CIP-191 (which names methods with strings/URNs), each tag has a documented URN alias. **These URNs do not appear in metadata** — they are a documentation/registry cross-walk, suitable for mirroring into the CIP-10 registry. The only place a URN legitimately appears in data is the `custom` type's `content_anchor` URI, which MAY name an external method schema.

| Tag | CIP-179 type | Interop URN | CIP-191 method |
|:----|:-------------|:------------|:---------------|
| 0 | Custom | (per-method, via the anchor URI) | — |
| 1 | Single-choice | `urn:cardano:poll-method:single-choice:v2` | `single-choice`, `binary` |
| 2 | Multi-select | `urn:cardano:poll-method:multi-select:v2` | `multi-choice` |
| 3 | Ranking | `urn:cardano:poll-method:ranking:v1` | `ranked` |
| 4 | Numeric-range | `urn:cardano:poll-method:numeric-range:v2` | `range` |
| 5 | Points-allocation | `urn:cardano:poll-method:points-allocation:v1` | `weighted` |
| 6 | Rating | `urn:cardano:poll-method:rating:v1` | `likert` |

CIP-191's `binary` method maps onto single-choice with explicit options; ranking/points-allocation/rating align with CIP-191's `ranked`/`weighted`/`likert`, which CIP-191 currently namespaces under `urn:ekklesia:*`.

**URN versioning.** The version suffix denotes a method's *semantic contract*, independent of this CIP's document version; it bumps only when a method's answer semantics change incompatibly. `single-choice`, `multi-select`, and `numeric-range` are at `:v2` because their `:v1` was defined by CIP-179 v1 (the string-based encoding) and is referenced by CIP-191; this document materially redefines them (CBOR-first encoding, abstain-by-omission, and — for multi-select — a meaningful empty selection), so the bump is intentional and CIP-191 would need to repoint at `:v2`. `ranking`, `points-allocation`, and `rating` have no prior `urn:cardano:poll-method:*` definition (CIP-191 carries them under `urn:ekklesia:*`), so they begin at `:v1` as their first definition under this namespace.

### Survey Definition

A survey definition is an integer-keyed CBOR map (deterministically encoded; see [Encoding Conventions](#encoding-conventions)):

| Key | Type | Description |
|:----|:-----|:------------|
| 0 | uint | Schema version. This document defines version `4`. |
| 1 | credential | Survey owner. Used to authorize cancellation. |
| 2 | chunked_text | Human-readable survey title. MAY be empty in external-content mode. |
| 3 | chunked_text | Human-readable survey context or rationale. MAY be empty in external-content mode. |
| 4 | eligible_roles | Non-empty set of roles permitted to respond. |
| 5 | uint | Inclusive epoch cutoff for response validity and tally snapshot reference. |
| 6 | submission_mode | How responses are submitted: `[0]` public, or `[1, ...]` timelocked. |
| 7 | array | Survey questions. MUST contain at least one item. |
| 8 | content_anchor | OPTIONAL external presentation document (URI + hash). Its presence signals external-content mode. |

Keys 0–7 are mandatory; key 8 is optional. Decoders SHOULD ignore unrecognized keys (reserved for future versions).

The survey definition transaction MUST prove ownership of the `owner` credential: for key-based credentials, the `addr_keyhash` MUST be in `required_signers`; for script-based credentials, tooling MUST verify the native script is satisfied (same rules as for response credentials).

### Question Types

Questions use a tagged sum type. The first element is the type tag. All question types include a `chunked_text` question prompt as their second element, and an OPTIONAL trailing `required` flag (default `false`). When `required` is `true`, a response MUST include an answer for that question (it MUST NOT be omitted); see [Abstain semantics](#abstain-semantics).

Option-bearing questions (single-choice, multi-select, ranking, points-allocation, rating) carry an `options_or_count`: either an inline array of at least 2 `bounded_text` labels, or — in external-content mode — a single `uint >= 2` giving the option count, with labels supplied by the external document. A CBOR decoder distinguishes the two forms directly (array vs. uint).

#### Custom (tag 0)

```
[0, question_prompt, content_anchor, ?required]
```

- `content_anchor`: `[method_schema_uri, method_schema_hash]`, where `method_schema_hash` is the `blake2b-256` of the raw bytes at `method_schema_uri`.
- Response uses `transaction_metadatum`, interpreted according to the referenced schema.
- Tag 0 is reserved for this extension type so that new built-in types can be appended at higher tags without renumbering it.

#### Single-choice (tag 1)

```
[1, question_prompt, options_or_count, ?required]
```

- Response MUST select exactly one option index. The index MUST be a valid position in the options.

#### Multi-select (tag 2)

```
[2, question_prompt, options_or_count, min_selections, max_selections, ?required]
```

- `max_selections`: positive integer; `min_selections`: non-negative integer; with `0 <= min_selections <= max_selections <= number_of_options`.
- Response MUST select between `min_selections` and `max_selections` option indices (inclusive). All indices MUST be valid positions and MUST NOT contain duplicates.
- When `min_selections` is `0`, a **present but empty** selection (`[2, q_idx, []]`) is a valid answer meaning "none selected" — for example, "which of these do you deem acceptable?" answered with nothing acceptable. This is distinct from omitting the question, which is an abstain (see [Abstain semantics](#abstain-semantics)). When `min_selections >= 1`, an empty selection is not a valid present answer; to not answer, omit the question.

#### Ranking (tag 3)

```
[3, question_prompt, options_or_count, min_ranked, max_ranked, ?required]
```

- `min_ranked`, `max_ranked`: positive integers with `1 <= min_ranked <= max_ranked <= number_of_options`.
- Response MUST provide between `min_ranked` and `max_ranked` option indices in preference order (most preferred first). Indices MUST NOT contain duplicates and MUST be valid positions.

#### Numeric-range (tag 4)

```
[4, question_prompt, [min_value, max_value, ?step], ?required]
```

- `min_value <= max_value`.
- Optional `step` MUST be a positive integer. When present, the response value MUST satisfy `(value - min_value) mod step == 0`.
- Response MUST contain an integer satisfying range and step constraints.

#### Points-allocation (tag 5)

```
[5, question_prompt, options_or_count, budget, ?required]
```

- `budget`: positive integer, the total number of points to distribute.
- Response is a list of `(option_index, points)` pairs: `points >= 0`, option indices unique and valid, and the sum of all `points` MUST equal `budget`. Options not listed receive `0`.

#### Rating (tag 6)

```
[6, question_prompt, options_or_count, rating_scale, ?required]
```

- `rating_scale` is either:
  - a `numeric_constraints` grid `[min_rating, max_rating, ?step]` defining a numeric scale (e.g. a 1–5 Likert scale), presented to respondents as numbers; or
  - an ordered list of at least 2 level labels from worst to best (e.g. `["bad", "average", "good"]`), presented to respondents as text. In external-content mode the labels MAY be a level count (`uint >= 2`), with the labels supplied by the external document.
- The rating value in a response is always an integer: for a numeric grid, a value on the grid; for a label list, the 0-based index into the labels (e.g. `0 = "bad"`, `2 = "good"`). This keeps tallying numeric and compact regardless of presentation.
- Response is a list of `(option_index, rating)` pairs: each `rating` valid for the scale, option indices unique and valid. A respondent MAY rate a subset of options.

### Abstain semantics

A response MAY answer a subset of questions. **A question for which a response carries no answer item is treated as an abstain on that question** (within that response). This makes abstention the default, costs zero bytes, and removes the need for a dedicated abstain encoding.

- Each question is tallied independently. For a given question, a submitted response either contributes an answer or counts as an abstain; tools SHOULD report the abstain count alongside the per-option tallies.
- A respondent who never submits any response to a survey is a non-participant, not an abstainer.
- If a question sets `required = true`, a response that omits it is invalid as a whole and MUST NOT be tallied. (Authors use `required` to force an explicit answer where abstention is not acceptable.)
- Authors who want abstain to be a *selectable, counted* option MAY still add an explicit "Abstain" label to an option-based question; that is distinct from the implicit abstain-by-omission above.
- Abstain (omission) is distinct from an explicit empty answer. A multi-select question with `min_selections = 0` accepts a present, empty selection (`[2, q_idx, []]`) meaning "none selected" — a deliberate answer that is tallied, unlike an omitted question. See [Multi-select (tag 2)](#multi-select-tag-2).

### Survey Response

A survey response is an integer-keyed CBOR map (deterministically encoded; see [Encoding Conventions](#encoding-conventions)):

| Key | Type | Description |
|:----|:-----|:------------|
| 0 | uint | Schema version. This document defines version `4`. |
| 1 | survey_ref | Reference to the survey: `[tx_id, survey_index]`. |
| 2 | role | Claimed responder role (integer). |
| 3 | credential | Responder's credential: `[0, addr_keyhash]` or `[1, script_hash]`. |
| 4 | response_answers | Plaintext answer items (public), or a chunked-bytes tlock ciphertext (timelocked). |
| 5 | content_anchor | OPTIONAL voter rationale (URI + hash). See [Voter Rationales](#voter-rationales). |

Keys 0–4 are mandatory; key 5 is optional. Decoders SHOULD ignore unrecognized keys (reserved for future versions).

The response carries its own `spec_version`. A response's `spec_version` MUST match that of the referenced survey definition. Including it lets tooling decode a response — in particular its answer encoding and whether answers are plaintext or timelocked ciphertext — without first resolving the survey.

Respondents MAY answer a subset of questions; omitted questions are abstains ([Abstain semantics](#abstain-semantics)). Question indices in the answers array MUST be unique within one response. Each answer's type tag MUST match the type tag of the referenced question.

A response transaction MUST NOT reference itself: the `tx_id` in `survey_ref` MUST differ from the response transaction's own id.

Multiple responses MAY be batched in a single transaction. Each is validated independently.

### Voter Rationales

A response MAY include an optional `content_anchor` (key 5) referencing an off-chain rationale document explaining the respondent's choices. The anchor is `[uri, blake2b-256 hash]`, making the rationale tamper-evident without paying for its bytes on-chain. This mirrors the off-chain rationale conventions used by Cardano governance (CIP-100 / CIP-108). The rationale is purely informational and does not affect validation or tallies.

### Survey Cancellation

A cancellation payload contains one or more `survey_ref` values, each identifying a survey to cancel.

Validation:
- The `survey_ref` MUST resolve to a previously published survey definition.
- The cancellation transaction MUST prove ownership of the survey definition's `owner` credential (same rules as for definition transactions: key-based via `required_signers`, script-based via native script satisfaction).
- A cancellation MUST be submitted before or during the survey's `end_epoch`. Cancellations with `cancellation_epoch > end_epoch` are invalid.
- Once cancelled, the survey is inactive as a whole. Tooling MUST NOT tally any responses for a cancelled survey.
- Cancellation does not remove the survey definition data from the chain; it signals that the survey should not be used.

### External Survey Definitions

For large surveys, the bulk of the bytes is human-readable text (title, description, prompts, option labels). A survey definition MAY move that text off-chain while keeping the full structural skeleton on-chain, by carrying a `content_anchor` (key 8).

In external-content mode:
- The text fields (`title`, `description`, and each question `prompt`) MAY be empty strings.
- Option-bearing questions MAY use the `option_count` form (`uint >= 2`) instead of inline labels.
- All structural and validation-relevant fields — question count, type tags, constraints (option counts, min/max selections, ranges, budgets, rating grids), `eligible_roles`, `end_epoch`, `owner`, `submission_mode` — remain on-chain.

The off-chain document referenced by the anchor supplies the missing presentation text: at minimum a title, description, and, per question (in definition order), a prompt and the option labels (in option-index order). Its `blake2b-256` hash MUST equal the anchor hash.

Properties:
- **Determinism preserved.** Responses are validated and tallied entirely from on-chain data, because answers reference option *indices* and every constraint is on-chain.
- **Graceful degradation.** If the off-chain document is unavailable, tooling can still validate responses and compute tallies; only labels cannot be rendered.
- **Tamper-evident.** The hash binds the off-chain text to the on-chain commitment.

This is deliberately *additive*: the on-chain structure is never replaced by a bare reference, so the survey is never uninterpretable for protocol purposes when the off-chain document is missing.

### Responder Identity and Deduplication

Identity verification leverages Cardano's existing transaction-level mechanisms. The responder's `credential` is included directly in the survey response, and the transaction proves ownership of that credential.

#### Credential proof

The `credential` in the response is a Conway ledger `credential`: `[0, addr_keyhash]` for key-based or `[1, script_hash]` for script-based.

Control of the response `credential` MUST be proven by **at least one** of the two mechanisms below. A response is valid if **either** holds; the two are alternatives, not cumulative.

**Mechanism A — `required_signers` (standalone or linked):**
- **Key-based credential** `[0, addr_keyhash]`: the `addr_keyhash` MUST be present in the transaction body's `required_signers` (field 14). The ledger enforces that the corresponding signature witness is present, proving ownership.
- **Native-script credential** `[1, script_hash]`: tooling MUST resolve the native script from the `script_hash` (via chain indexing) and verify that the keys present in the transaction's `required_signers` satisfy the script's conditions.
- **Plutus-script credential** `[1, script_hash]`: not provable this way — a Plutus script needs a redeemer to be evaluated, and metadata has no redeemer tag.

**Mechanism B — governance-vote binding (governance-linked surveys only):** the transaction carries a `voting_procedures` entry whose voter credential equals the response `credential` and votes on `linked_action_id`. The ledger already enforces that this voter witnessed the transaction — a key witness for a key-based voter, native-script satisfaction or Plutus redeemer evaluation for a script voter — so the binding **is** the credential proof on its own.

- When mechanism B is satisfied, the credential need **not** also appear in `required_signers`; B alone is sufficient for key-based, native-script, and Plutus-script credentials.
- For **Plutus-script** credentials, B is the *only* available mechanism (A cannot prove them); see [Plutus script credentials](#plutus-script-credentials).
- B is otherwise optional: key-based and native-script credentials may use A, B, or both. Using B additionally ties the response to an on-chain governance vote. The presence-conditional cross-checks for B (voter-credential match, vote on the linked action, voter-tag vs. claimed role) are in [Linked survey response rules](#linked-survey-response-rules).

#### Role validation

The claimed role MUST be validated against ledger state:

| Role | Ledger requirement |
|:-----|:-------------------|
| DRep (0) | `credential` MUST be a registered DRep credential. |
| SPO (1) | `credential` MUST be the cold credential of a registered pool operator. |
| CC (2) | `credential` MUST be the hot credential of an active Constitutional Committee member. |
| Stakeholder (3) | `credential` MUST be a stake credential with delegated stake. |
| Owner (4) | `credential` MUST be a payment/spending credential. |

Tools MUST NOT trust role claims without validation. A response is invalid when the claimed role does not match ledger-derived role evidence (for roles that have such evidence).

A signer MAY submit separate responses for different roles, provided each role claim independently passes validation.

> **SPO via calidus (open).** Allowing SPOs to prove the SPO role with a CIP-151 calidus hot key, rather than the pool cold credential, is desirable but may depend on the CIP-8 proof design pass (see [Limitations and Future Extensions](#limitations-and-future-extensions)). Until that lands, the cold-credential path above is the defined one.

#### Validation phases

Validation runs in two phases:
1. **Response-time validation**: at response inclusion, verify credential proof and role membership.
2. **Tally-time re-verification**: at the survey's `end_epoch` snapshot, re-check role membership and credential eligibility.

Only responses that pass both phases are counted in tallies.

#### Deduplication

The identity tuple for deduplication is `(survey_ref, role, credential)`.

If multiple responses from the same tuple pass both validation phases, the latest valid response wins.

### Governance Action Linkage

Survey-to-action linkage is canonicalized as **Action -> Survey** and is restricted to **Info Actions**. Info Actions are the only governance action type guaranteed to run their full validity period without on-chain side effects, making them suitable for structured sentiment gathering.

Linkage is primarily a **discovery/advertisement** relationship (the Info Action points readers at the survey) together with an **epoch-alignment** invariant (the survey closes exactly when the action's vote does). It does **not** restrict who may respond: every role in the survey's `eligible_roles` participates in a linked survey exactly as it would standalone. Casting a governance vote on the linked action is an *optional* enrichment for the voter roles, not a precondition for responding (see [Linked survey response rules](#linked-survey-response-rules)).

When an Info Action links to a survey, the governance action anchor metadata MUST include:

```json
{
  "specVersion": 4,
  "kind": "cardano-governance-survey-link",
  "surveyTxId": "<hex-encoded 32-byte transaction id>",
  "surveyIndex": 0
}
```

(The anchor metadata is an off-chain document; `surveyTxId` is hex-encoded per JSON convention.)

Validation rules:
- The governance action MUST be an Info Action. Linkage to any other governance action type is invalid.
- The `(surveyTxId, surveyIndex)` pair MUST resolve to a transaction that includes label `17` with a survey definitions payload, and the survey at the given index MUST exist.
- Tooling MUST derive the linked governance action id (`linked_action_id`) from the Info Action that carries the survey-link anchor.
- Linkage does not subset eligibility: all roles in the survey's `eligible_roles` may respond. (The optional governance-vote binding below is available only to the roles with a Conway voter type — `{DRep (0), SPO (1), CC (2)}` — but the other roles still participate via the standalone proof.)
- Survey `end_epoch` MUST exactly match the Info Action's active voting end epoch. If they differ, the link is invalid.
- If linkage validation fails, tooling MUST NOT attach that survey to the Info Action.
- Linkage invalidity does not invalidate the survey as standalone metadata.

#### Conway voter tag to CIP-179 role mapping

For governance-linked surveys, the Conway `voter` type in `voting_procedures` maps to CIP-179 roles as follows:

| Conway voter tag | Conway meaning | CIP-179 role |
|:-----------------|:---------------|:-------------|
| 0 | CC hot key credential | CC (2) |
| 1 | CC hot script credential | CC (2) |
| 2 | DRep key credential | DRep (0) |
| 3 | DRep script credential | DRep (0) |
| 4 | SPO pool cold key credential | SPO (1) |

#### Linked survey response rules

A response to a governance-linked survey is validated exactly like a standalone response — credential proof plus role validation — and MUST be accepted on that basis alone, for every eligible role. A linked response is **not** required to carry a `voting_procedures` entry.

A response MAY additionally **bind** itself to the linked governance vote by including a `voting_procedures` entry. This binding ties the survey answer to an on-chain governance vote on the same action, and is the only mechanism that supports **Plutus-script credentials** (the ledger evaluates the voting redeemer; the standalone `required_signers` path cannot — see [Plutus script credentials](#plutus-script-credentials)). When such a binding is present:
- The `voting_procedures` MUST contain a voter entry whose credential matches the response's `credential`.
- That voter entry MUST include a vote on `linked_action_id`.
- The role derived from the matching voter entry (per the Conway voter tag mapping above) MUST match the claimed role.
- The claimed role is necessarily one of `{DRep, SPO, CC}`, the only roles with a Conway voter type. Stakeholder and Owner responses cannot use this binding and rely on the standalone proof.

When the binding is present and these checks pass, it **satisfies credential proof on its own** (mechanism B in [Credential proof](#credential-proof)): the same credential does **not** also need to appear in `required_signers`, because the ledger already enforced the voter's witness when it accepted the vote. A binding that is present but fails any of the checks above makes the response invalid; an *absent* binding is not a failure.

### Epoch Semantics

- `end_epoch` is mandatory in every survey definition and MUST be greater than the current epoch at the time of the definition transaction.
- `response_epoch` is derived from the response transaction's inclusion slot via ledger epoch mapping.
- Responses with `response_epoch > end_epoch` are invalid.
- Role-membership and credential eligibility checks are evaluated at response time and re-verified at tally time using the `end_epoch` snapshot.
- A response is counted only if it passes both response-time validation and `end_epoch` tally-time re-verification.
- `end_epoch` also serves as the canonical snapshot epoch that any downstream weighting/aggregation process (out of scope here) can reference.

### Duplicate and Ordering Semantics

For a given identity tuple `(survey_ref, role, credential)`:
- If multiple responses pass both validation phases, the latest valid response wins.
- Latest is determined by the chain ordering tuple `(slot, tx_index_in_block, response_index)`, where `response_index` is the position within the responses array in the label `17` payload.
- Latest-response semantics replace the full prior response for that tuple.

### Weighting and Aggregation (out of scope)

This CIP records *who responded, in what role, and with what selection*. It deliberately does **not** standardize how a recorded selection is weighted or aggregated into an outcome.

- The infrastructure treats every valid response as one recorded response. Whether results are counted one-per-credential, by stake, by pledge, quadratically, by Borda count, or under any other scheme is decided by the consumer of the recorded vote set, outside this CIP's boundaries.
- Weighting does not change what a respondent signs or how a response is encoded; it changes only the interpretation of results. Keeping it out of scope means the same recorded set can be re-tallied later under a different rule, and avoids baking an incomplete enumeration of weighting modes (which cannot express schemes such as quadratic voting) into the on-chain format.
- `role` therefore expresses eligibility and a UI key-selection hint only — not a weighting mode.

### Tool Output Requirements

- Tools SHOULD expose per-role participation tallies as primary outputs whenever multiple roles are eligible, including the per-question abstain counts.
- Tools MUST NOT present any weighted or merged total as if it were produced or sanctioned by this CIP. Any weighting/aggregation applied downstream MUST disclose its policy and snapshot basis.
- Audit/export output SHOULD include role, `credential`, counted/excluded status, and exclusion reason when applicable.
- A canonical, machine-comparable tally interchange format (so independent tools can confirm matching results) is deferred; see [Limitations and Future Extensions](#limitations-and-future-extensions).

### Transaction-level Constraints

- A single label `17` payload MUST contain exactly one of: definitions, responses, or cancellations.
- A response transaction MUST NOT reference itself.
- If Info Action linkage is provided, it MUST be encoded in the Info Action's anchor metadata, not in the survey definition.

### Block Explorer and dApp Implementation Guide

1. Discover survey definitions by scanning label `17` for payloads with tag `0`.
2. Discover cancellations (tag `2`) and mark cancelled surveys as inactive.
3. Optionally discover Info Actions with anchor metadata carrying `kind = "cardano-governance-survey-link"`.
4. If present, validate that the governance action is an Info Action, then validate linkage by `(surveyTxId, surveyIndex)` and exact `end_epoch` equality; derive `linked_action_id`. (Linkage does not restrict which roles may respond.)
5. Discover responses by scanning label `17` for payloads with tag `1`.
6. Resolve each response to its survey via `survey_ref`.
7. Reject responses to cancelled surveys.
8. Validate each answer against the corresponding question type and constraints. Treat any question without an answer item as an abstain (reject the whole response if a `required` question is omitted).
9. Verify each response's `credential` by **either** mechanism: (A) `required_signers` — key presence (key-based) or native-script satisfaction (native-script); **or** (B) a ledger-validated `voting_procedures` binding for that credential (the only path for Plutus-script credentials, and sufficient on its own — when B holds, do not also require the credential in `required_signers`). When a binding is present, cross-check the Conway voter tag against the claimed role. Then validate role membership against ledger state.
10. Filter responses by `response_epoch <= end_epoch`.
11. At or after `end_epoch`, re-verify each response using `end_epoch` snapshot state; exclude failures.
12. Apply latest-valid-response-wins per `(survey_ref, role, credential)`.
13. Produce per-role participation tallies and per-question abstain counts. Any weighting/aggregation is a downstream, out-of-scope step.
14. If the definition carries a `content_anchor`, fetch and hash-verify the external document to render presentation text; validation and tallying do not depend on it.

## Rationale: How does this CIP achieve its goals?

The goal is a compact, deterministic, and interoperable on-chain survey format that integrates naturally with Cardano's existing infrastructure. Each design decision follows from that goal.

### Compact on-chain encoding

On-chain metadata is paid for per byte in transaction fees and stored permanently by every full node. The encoding follows the same conventions as the Cardano ledger CDDL: integer map keys, integer enum tags, and raw bytes for hashes. This avoids the overhead of text-based keys and values, which would multiply the cost of every survey definition and every response.

### Chunked text for human-readable fields

Cardano metadata limits individual text strings to 64 bytes. Survey titles, descriptions, and question prompts may exceed this limit. When the value fits within 64 bytes, it is stored as a plain `bounded_text`. When it exceeds 64 bytes, chunked text arrays (as established by CIP-20) are used. Implementations must accept both forms. Option labels remain plain `bounded_text` since they are typically short identifiers.

### Tagged sum types for questions and answers

Each question type has distinct required fields. A tagged sum type makes invalid combinations unrepresentable in the schema: a single-choice question cannot carry numeric constraints, and a numeric-range question cannot carry options. This moves validation from prose rules into the data model, reducing the burden on implementors.

### Custom type at tag 0

The custom (extension) type is placed at tag `0` so it occupies a fixed, stable position regardless of how many built-in types are later appended at higher tags. New built-in types never disturb the extension point, and the catch-all has a predictable home for decoders.

### Points-allocation and rating types

Budget-style distribution (`points_allocation`) and discrete rating scales (`rating`) are common community-polling needs that the option-only types cannot express, and they align with CIP-191's `weighted` and `likert` methods. Adding them as their own tags keeps the surface compatible with that standard.

### Abstain by omission

Abstention is the most common non-answer and benefits from a zero-byte encoding. Treating an omitted question as an abstain avoids a dedicated abstain marker on every skipped question while still allowing creators to force an explicit answer with `required`, and authors to offer an explicitly counted "Abstain" option when they want one.

### Off-chain content anchors

A single `content_anchor` primitive (URI + `blake2b-256` hash) serves three needs: bulky survey presentation text, custom-method schemas, and voter rationales. It keeps the on-chain footprint small and tamper-evident, while — for survey definitions — preserving full on-chain validation and tally determinism because only presentation text moves off-chain (structure and constraints stay on-chain).

### Survey references as (TxId, index)

On-chain artifacts in Cardano use `(TxId, index)` pairs for identification: UTxOs use `transaction_input = [transaction_id, index]`, governance actions use `gov_action_id = [transaction_id, gov_action_index]`. Survey references follow the same pattern, which enables batching multiple definitions in a single transaction and aligns with tooling that already handles this identification scheme.

### Batched definitions and responses

Multiple survey definitions or responses per transaction reduce the number of transactions needed, directly lowering fees for survey creators and active respondents.

### Survey cancellation

A survey with errors wastes respondent effort. A simple cancellation mechanism (referencing the survey and proving ownership of the `owner` credential) lets creators invalidate a broken survey without waiting for `end_epoch`. The `owner` is explicit in the survey definition, so cancellation authorization is unambiguous.

### Explicit credential in response

The responder's `credential` is included directly in the survey response metadata. Verification is straightforward: for key-based credentials, the `addr_keyhash` must be in `required_signers` (ledger-enforced signature check); for script-based credentials, tooling resolves the native script via chain indexing and verifies that the transaction's required signers satisfy its conditions. For governance-linked surveys, `voting_procedures` provides the same guarantee. Including the credential in the response eliminates ambiguity (no need to guess which signer is the responder) and supports native multisig scripts in standalone surveys.

### Multi-question surveys with partial responses

Grouping related questions in one survey avoids unnecessary proliferation of survey transactions while keeping shared constraints (`eligible_roles`, `end_epoch`) at the survey level. Allowing partial responses gives respondents flexibility; each question is tallied independently over the responses that include it, with omitted questions counted as abstains.

### Latest-valid-response-wins

Participants may want to change their answer before `end_epoch`. Accepting the latest valid response per identity tuple provides a correction path while maintaining deterministic, reproducible tallies.

### Eligibility as a role set, weighting out of scope

Earlier drafts paired each role with a weighting mode (`CredentialBased` / `StakeBased` / `PledgeBased`). That conflated two separable concerns: *who may respond* (eligibility, which affects validation) and *how results are interpreted* (weighting, which affects nothing the respondent signs). Recording only an eligibility role set keeps the on-chain definition honest about what the chain can enforce, supports a UI key-selection hint per role (e.g. an `Owner` presents a spending key), and lets any weighting scheme — including ones the old enumeration could not express, such as quadratic voting — be applied downstream against the same recorded vote set.

### Info Action linkage

Governance linkage is restricted to Info Actions because they are the only governance action type guaranteed to complete their full validity period without on-chain side effects. The action anchor references the survey (not the other way around), avoiding circular transaction dependencies: the survey is published first, then the Info Action references it by `survey_ref`.

Linkage is treated as discovery plus epoch alignment, not as an eligibility gate. Earlier drafts required every linked response to carry a `voting_procedures` vote on the linked action, which made a governance vote the proof mechanism and therefore silently excluded the two roles without a Conway voter type (Stakeholder and Owner). That conflated advertising a survey with restricting who could answer it. Since the standalone credential proof and ledger role-validation already cover every role — including DReps, SPOs, and CC — the governance-vote binding is kept as an *optional* enrichment rather than a requirement. Retaining it (rather than dropping it entirely) is deliberate: it is the only mechanism that yields ledger-evaluated verification of Plutus-script credentials, which `required_signers` cannot provide.

## Limitations and Future Extensions

### CIP-8 message-signing proof (and calidus support)

`required_signers` proves a credential's key witnessed the transaction, which is ledger-enforced and simple. It does not, however, cover credential types that sign messages rather than transactions — most notably CIP-151 calidus hot keys, which would let SPOs prove the SPO role without their cold credential. An optional CIP-8 (`COSE_Sign1`) proof mode could enable this, and additionally allow a third party to batch-submit many message-signed responses in one transaction. Before specifying it, a design pass must settle: exactly what payload is signed (binding at least `survey_ref` to prevent cross-survey replay, plus a per-response nonce), where the signature and recovered key live in the response, how replay within a survey is handled (the existing `(survey_ref, role, credential)` dedup helps), and how batched submission is expressed in the label `17` payload. Tracked as an open design question.

### Canonical tally interchange format

CIP-191 commits a fully-specified `results.json` (hashed on-chain, re-tallied by auditors), which forces a rigid per-method tally schema. CIP-179 derives tallies independently from on-chain data and commits no tally artifact, so it does not need that rigidity. A *recommended, non-normative* interchange shape for per-method tallies (option counts, numeric distributions, ranking first-preferences and pairwise matrices, points and rating aggregates, plus per-role and abstain breakdowns) would still help independent tools confirm matching results for shared test vectors. Designing it is deferred to its own pass.

### Plutus script credentials

Credential proof for a survey response depends on the credential type:

- **Key-based** and **native-script** credentials are proven via `required_signers` (directly for keys; by satisfying the resolved native script for native scripts). This works in both standalone and governance-linked surveys.
- **Plutus-script** credentials cannot be proven this way: a Plutus script must be evaluated against a redeemer, and transaction metadata has no redeemer tag. Consequently there is **no standalone path** for a Plutus-script credential.

The one supported path for a Plutus-script credential is the **optional governance-vote binding** of a governance-linked survey: the Conway `voter` type in `voting_procedures` supports Plutus script-based voters (tags 1 and 3), and the ledger evaluates the voting redeemer for that voter. A linked response carrying such a binding therefore gets ledger-evaluated Plutus verification "for free." This is a concrete reason the optional binding is retained rather than dropped when linkage was relaxed to pure discovery.

A future version could define an alternative mechanism (e.g. a dedicated witness convention) to verify Plutus-script credentials for standalone surveys; none exists today.

### Versioning granularity

This standard uses a single integer (`spec_version = 4`). Encoding the two top-level records as integer-keyed maps already absorbs the most common kind of backward-compatible extension: a new optional field can be added at a fresh key, and decoders that ignore unrecognized keys keep working without a version bump. If finer-grained signalling of such additions proves necessary, a more granular scheme (e.g., `[major, minor]`) could be adopted. For the scope of this standard, a single integer is sufficient: any breaking change increments the version and defines the new layout.

### Version history

Version 2 extends version 1 with the following breaking changes. Integer keys and enumeration values replace string-based encoding throughout, addressing Cardano's 64-byte metadata text limit and reducing per-byte cost. A ranking question type is added to the tagged sum type. Survey identification uses (TxId, survey_index) pairs, enabling multiple definitions or responses to be batched in a single transaction. A cancellation payload type allows the survey owner to invalidate a survey before or during its end_epoch. Credential proof moves from explicit hex inclusion to verification via required_signers and native script satisfaction.

Version 3 extends version 2 with the following breaking changes. A `submission_mode` field is added to `survey_definition` as a tagged sum type (`[0]` public, `[1, chain_hash, round, padding_size]` timelocked), enabling timelocked responses via Drand `tlock` (delayed reveal, not permanent secrecy). The `survey_definition` array is reordered so fixed-size header fields precede the variable-length `questions` array. A `spec_version` field is restored as the first element of `survey_response`.

Version 4 extends version 3 with the following breaking changes:
- **Question type tags renumbered; custom moves to tag 0.** The extension type now occupies a stable tag `0`; built-in types follow (single-choice 1, multi-select 2, ranking 3, numeric-range 4), and two new types are appended.
- **Two new question types:** `points_allocation` (tag 5, budget distribution) and `rating` (tag 6, discrete rating scale — numeric grid or ordered text labels such as "bad"/"average"/"good", always encoded as an integer), aligning with CIP-191's `weighted` and `likert` methods.
- **`min` constraints added** to multi-select (`min_selections`) and ranking (`min_ranked`); both `min`/`max` are now explicit positional fields. Multi-select allows `min_selections = 0`, so a present empty selection ("none selected") is a counted answer distinct from abstaining; ranking requires `min_ranked >= 1`.
- **Per-question `required` flag** (optional trailing bool, default false). With abstain-by-omission, `required` forces an explicit answer.
- **Abstain by omission.** A question with no answer item in a response is an abstain on that question; the empty-multi-select special case is removed.
- **`role_weighting` replaced by `eligible_roles`; weighting modes removed.** Weighting/aggregation is moved entirely out of scope, because it is purely interpretive (it does not change what a respondent signs), an enumerated set of modes cannot express schemes such as quadratic voting, and the same recorded set should be re-talliable under any rule. A new `Owner` role (control of a payment/spending credential; off-chain eligibility gating) is added.
- **`content_anchor` primitive (URI + hash)** introduced and reused for: optional external survey presentation text (enabling much larger surveys at small on-chain cost while keeping structure/validation on-chain), the custom-method schema reference, and an optional per-response voter rationale.
- **Method Identifier Registry** added: a documentation/registry URN cross-walk for the integer tags (URNs do not appear on-chain), aligning with CIP-191. The `single-choice`, `multi-select`, and `numeric-range` URNs are bumped to `:v2` (their `:v1`, from CIP-179 v1's string-based encoding, is referenced by CIP-191 and is superseded here); `ranking`, `points-allocation`, and `rating` begin at `:v1` as new `urn:cardano` definitions.
- **Top-level records encoded as integer-keyed maps.** `survey_definition` and `survey_response` move from positional arrays to deterministically-encoded integer-keyed CBOR maps (mirroring Conway `transaction_body`), so future optional fields can be added at new keys without renumbering or breaking decoders. The nested tagged unions (questions, answers) stay positional arrays.
- **Governance linkage no longer restricts eligibility.** A linked survey now admits all of its `eligible_roles` (previously only `{DRep, SPO, CC}` could respond, and every linked response was required to cast a governance vote on the action). The `voting_procedures` vote becomes an *optional* binding — retained because it is the only path that supports ledger-evaluated Plutus-script credential verification — rather than a precondition. Stakeholder and Owner can now participate in linked surveys via the standalone proof.
