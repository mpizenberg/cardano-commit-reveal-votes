# CIP-179 Demo

This is a stripped down example from the CF voting app to something dedicated to demo CIP-179 (Surveys and Polls).

## Running it

Compile the Elm source and serve the `static/` directory:

```sh
elm-cardano make src/Main.elm --debug --output static/main.js
# Then serve static/ with any HTTP server, for example:
python -m http.server 3000 --directory static
```

Open `http://localhost:3000` in a browser.

## File structure

```
minimal/
  elm.json                    -- Elm project config, references ../frontend/elm-cardano/src
  src/
    Main.elm                  -- App entry point: ports, model, update, view
    Api.elm                   -- HTTP requests to Koios/Ogmios APIs
    ProposalMetadata.elm      -- CIP-108 metadata decoder (copied from frontend)
  static/
    index.html                -- HTML shell with inline CSS and JS bootstrap
    elm-cardano.js            -- CIP-30 wallet JS companion (generated by elm-cardano)
    elm-concurrent-task.js    -- Async task runner (from andrewMacmurray/elm-concurrent-task)
    storage.js                -- IndexedDB wrapper
    pkg-uplc-wasm/            -- UPLC WASM module for Plutus script evaluation
```

## How it works

### Initialization sequence (from previous codebase)

Everything starts in `static/index.html`. The JS bootstrap runs in this order:

1. **UPLC WASM init** -- `elm-cardano.js` imports and initializes the UPLC WebAssembly module
   (`pkg-uplc-wasm/`), exposing it on `window.uplc_wasm`. This is needed by elm-cardano
   for Plutus script evaluation, even if this minimal app does not build transactions.

2. **Elm app compilation loaded** -- `ElmCardano.loadMain("/main.js")` injects the compiled
   Elm bundle as a `<script>` tag. This must happen after WASM init so `window.uplc_wasm`
   is available.

3. **IndexedDB initialized** -- `Storage.init()` opens (or creates) an IndexedDB database
   with a `proposalMetadata` object store for caching fetched metadata.

4. **Elm app started** -- `Elm.Main.init()` receives three flags:
   - `url` -- the current page URL (unused in this minimal version, kept for compatibility)
   - `db` -- the IndexedDB connection handle (an opaque JS value passed through Elm as `Json.Decode.Value`)
   - `networkId` -- `0` for Preview/Testnet, `1` for Mainnet

5. **Wallet ports wired** -- `ElmCardano.init()` subscribes to the `toWallet` port and
   connects responses back through `fromWallet`. This enables CIP-30 wallet communication.

6. **Task ports registered** -- `ConcurrentTask.register()` connects the `sendTask`/`receiveTask`
   ports and registers two custom task handlers (`storage:read`, `storage:write`) that
   the Elm `Storage` module calls.

### Elm-side initialization (Main.elm `init`)

When the Elm app starts, it fires three commands in parallel:

- **Load protocol parameters** from the Koios Ogmios proxy endpoint
  (`queryLedgerState/protocolParameters`). Retrieves Plutus cost models.
- **Query current epoch** from Koios (`queryLedgerState/epoch`).
  Once the epoch is known, proposals are fetched.
- **Discover CIP-30 wallets** by sending a `discoverWallets` request through the `toWallet` port.
  The JS side inspects `window.cardano` for installed browser wallet extensions.

### Data flow: proposals

```
init
  |
  +--> queryEpoch -----> GotEpoch
                            |
                            +--> loadGovProposals -----> GotProposals
                                                            |
                                  for each proposal:        |
                                  taskLoadProposalMetadata --+--> GotProposalMetadata
                                  (with IndexedDB caching)
```

1. **Epoch** is fetched via a POST to the Koios Ogmios endpoint.
2. **Proposals** are fetched from the Koios `/proposal_list` REST endpoint,
   filtered to only include proposals expiring after the current epoch.
   Proposals that have been ratified or expired are also filtered out on the Elm side.
3. **Metadata** for each proposal is fetched asynchronously using `elm-concurrent-task`.
   For IPFS URLs (`ipfs://...`), requests are staggered across six IPFS gateways --
   the first fires immediately, each subsequent one fires 2 seconds later,
   and the first successful response wins (via `ConcurrentTask.race`).
   Results are cached in IndexedDB keyed by the on-chain metadata hash,
   so subsequent page loads skip the IPFS fetch entirely.

### Data flow: wallet

```
init
  |
  +--> discoverWallets --[toWallet port]--> JS inspects window.cardano
                                              |
                          [fromWallet port]<--+-- AvailableWallets [...]
                                                    |
  user clicks "Connect X" -----> enableWallet --[toWallet port]--> JS calls wallet.enable()
                                                                      |
                                  [fromWallet port]<------------------+-- EnabledWallet wallet
```

Wallet communication uses two Elm ports (`toWallet` / `fromWallet`) that connect to
the `elm-cardano.js` companion file:

- **Discovery**: Elm sends a `cip30-discover` request. JS iterates `Object.keys(window.cardano)`,
  reads each wallet's name, icon, API version, and supported extensions,
  and sends back an `AvailableWallets` response.
- **Connection**: When the user clicks a wallet button, Elm sends a `cip30-enable` request
  with the wallet ID and requested extensions (CIP-95 for governance if supported).
  JS calls `window.cardano[id].enable()` and returns an `EnabledWallet` response.

This minimal app discovers and connects wallets but does not use the wallet API further
(no UTxO fetching, no transaction signing).

### Elm-JS interop: ports

| Port | Direction | Purpose |
|------|-----------|---------|
| `toWallet` | Elm -> JS | Send CIP-30 wallet requests (discover, enable) |
| `fromWallet` | JS -> Elm | Receive wallet responses |
| `sendTask` | Elm -> JS | Send concurrent task requests (storage, HTTP) |
| `receiveTask` | JS -> Elm | Receive task results |

The wallet ports carry `Json.Decode.Value` and are encoded/decoded using
`Cardano.Cip30.encodeRequest` and `Cardano.Cip30.responseDecoder` from the elm-cardano library.

The task ports are managed by `elm-concurrent-task` and carry batched async operations.
The `Storage` module uses them to read/write IndexedDB via the custom task handlers
registered in `index.html`.

### Timelocked ballots and padding

A survey can be created with **timelocked ballots**: each response's answers are
encrypted at submission with Drand `tlock` (quicknet) and only become decryptable
once a chosen Drand round publishes. This is a *delayed reveal*, not permanent
secrecy -- after the round, anyone can decrypt every ballot. The crypto runs in
JS (`static/tlock.js`) and is reached from Elm through two `elm-concurrent-task`
tasks (`tlock:encrypt` / `tlock:decrypt`, see `Tlock.elm`).

**Why padding.** The encrypted blob's length leaks information: a longer
ciphertext means a longer plaintext, which can reveal *how much* a voter
answered (e.g. how many options they selected). To avoid this, the CBOR-encoded
answers are zero-padded up to a fixed `paddingSize` before encryption
(`Survey.plaintextHexForAnswers`). Decryption ignores the trailing zeros because
the CBOR answer array is self-delimiting. The padding size is stored once per
survey in its on-chain metadata, so every ballot for that survey shares it.

**Choosing the size automatically.** Leaving the "Padding size" field blank uses
`Survey.maxPlaintextSize`, the worst-case CBOR size of a *fully answered* ballot
for that survey's questions. Padding every ballot up to this value makes all
ciphertexts the same length, so size leaks nothing. The estimate sums a
per-question upper bound using the standard CBOR integer-width rules
(0..23 -> 1 byte, 24..255 -> 2, ..., capped at the 64-bit form, 9 bytes):

- **Single / multi / ranking**: choice indices are bounded by the option count;
  list answers by the max-selections / max-ranked limit.
- **Numeric**: the wider of the two range bounds, with negatives sized by the
  CBOR magnitude `-1 - v`, capped at 64-bit.
- **Free text (`Custom`)**: counted as the empty string `""`.

**Limitation -- free text.** Because a free-text answer has no length bound, the
estimate counts it as empty. A survey with a `Custom` question therefore has no
size-hiding guarantee for that question: a real free-text answer longer than the
padding produces a longer ciphertext, revealing that the voter wrote something.
Bounding it would require imposing an explicit maximum answer length. The other
four question types are fully bounded, so for surveys without free text the
auto-sized padding makes every ciphertext identical in length.

`tests/SurveyTests.elm` validates this: it builds a maximal ballot, encodes it
through the real CBOR encoder, and checks the actual width never exceeds
`maxPlaintextSize` (run with `elm-test`).

### Elm modules

**Main.elm** -- The entire application in one file. Defines:
- `Flags`, `Model`, `Msg` types
- `init` with the three startup commands
- `update` handling epoch, proposals, metadata, and wallet responses
- `view` rendering a plain HTML list of proposal cards
- Port declarations for wallet and task communication

**Api.elm** -- All HTTP requests:
- `loadProtocolParams` -- POST to Koios Ogmios for Plutus cost models
- `queryEpoch` -- POST to Koios Ogmios for current epoch
- `loadGovProposals` -- GET from Koios `/proposal_list`
- `taskLoadProposalMetadata` -- ConcurrentTask fetching metadata from IPFS or HTTPS

**ProposalMetadata.elm** -- Decodes CIP-108 proposal metadata JSON:
- Extracts `body.title`, `body.abstract`, and `authors` list
- Computes a Blake2b-256 hash of the raw metadata for on-chain hash verification

**Storage.elm** -- Thin wrapper over `elm-concurrent-task` custom tasks:
- `read` / `write` -- IndexedDB operations via the `storage:read` / `storage:write` task handlers
- `cacheWrap` -- Decorator that checks IndexedDB first, falls back to a task, then caches the result

### Dependencies

This app reuses the `elm-cardano` library source from `../frontend/elm-cardano/src`
(referenced in `elm.json` source-directories). It provides:
- `Cardano.Cip30` -- CIP-30 wallet protocol encoding/decoding
- `Cardano.Gov` -- Governance types (ActionId, Vote, CIP-129 bech32 encoding)
- `Cardano.Address` -- Network ID types
- `Bytes.Comparable` -- Byte string utilities with hex encoding and Blake2b hashing

Key Elm package dependencies:
- `andrewMacmurray/elm-concurrent-task` -- Composable async tasks with port-based JS interop
- `krisajenkins/remotedata` -- `WebData` type for representing loading states
- `elm-toulouse/cbor` -- CBOR encoding (used internally by elm-cardano)
- `elm-cardano/bech32` -- Bech32 encoding for Cardano addresses and governance IDs
